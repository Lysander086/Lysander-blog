# システム設計面接の回答方法（完全ガイド）

---

## システム設計面接とは

- 複雑な問題を解決するためのシステムやアーキテクチャを設計する能力を評価する面接
- **100%完璧な解答を求めているわけではない**
- 評価されるポイント:
  - アーキテクチャの設計図を作成する能力
  - 複雑な問題を分析する能力
  - 複数の解決策を議論し、メリット・デメリットを明確に比較する能力
  - 実用的な解決策にたどり着く能力
- コーディング面接とは異なり、**オープンエンドな設計問題**が出題される
- 所要時間: 約**45分**（冒頭の自己紹介5分 + 最後の質問タイムを含む）

---

## フレームワーク: 5つのステップ

### ステップ1: 問題空間を定義する

**目的**: 問題を理解し、設計のスコープを定義する

- [ ] たくさん質問して、問題のスコープを「広く曖昧」→「狭く具体的」に絞り込む
- [ ] **機能要件**と**非機能要件**を明確にする
- [ ] スコープの内外（何を含み、何を含まないか）を定義する
- [ ] 前提条件と判断を面接官に共有する（面接官をガイドツアーに連れていくイメージ）

#### 確認すべきポイント
- ゼロからシステムを構築するのか？
- クライアントや利用者は誰か？
- 既存システムとの連携は必要か？既存のコンポーネントは何か？

#### 非機能要件の例
| 項目                | 説明                    |
| ----------------- | --------------------- |
| 可用性（Availability） | システムがどの程度常に利用可能であるべきか |
| 一貫性（Consistency）  | データの整合性をどこまで保つか       |
| 速度（Speed）         | レスポンスタイムの要件           |
| セキュリティ（Security）  | セキュリティ要件              |
| 信頼性（Reliability）  | 障害耐性                  |
| コスト（Cost）         | 予算制約                  |

> **ヒント**: 制約が多い場合は、最も重要なものに集中する。面接官が特定の方向に誘導している場合は、そのヒントに従う。すべての要件を完璧にしようとして時間を使い過ぎない。

### ステップ2: ハイレベル設計を行う

**目的**: システムの基本的な構成要素を示し、それらがどう連携するかを説明する

- [ ] **APIを設計する**: クライアントがシステムのリソースにアクセスするための契約
  - 各システム要件は1つ以上のAPIに対応させる
  - APIの種類を選択し、**理由を説明する**:

| API種類 | 特徴 |
|---------|------|
| REST | リソースベース、広く使われている |
| SOAP | XMLベース、エンタープライズ向け |
| GraphQL | クライアントが必要なデータだけ取得可能 |
| RPC | 関数呼び出し形式 |

- [ ] リクエストパラメータとレスポンスタイプを検討する
- [ ] クライアントとWebサーバーの通信方法を考える
- [ ] **ハイレベル設計図（ダイアグラム）を作成する**
  - 設計の設計図として機能し、機能要件を満たすための最も重要な部分を強調する
  - データフローと制御フローを示す

> **注意**: この段階ではまだスケーラビリティに踏み込まない（それは後のステップ）

---

### ステップ3: 設計を深掘りする

**目的**: システムのコンポーネントと関係をより詳細に検討する

- [ ] 非機能要件が設計の選択にどう影響するかを説明する
- [ ] 面接官が特定の領域に焦点を当てるよう促す場合もあるが、**会話の主導権は自分が持つ**

#### 考慮すべきポイントの例

| シナリオ | 対応策 |
|---------|--------|
| オンラインシステムで最新データが必要 | データ取り込み・処理・クエリプロセスの高速化 |
| 保存するデータ量が大きい | データベースのパーティショニングでストレージとクエリトラフィックを分散 |
| 読み書きトラフィックが多い | ロードバランサー層の導入 |

> **最も重要なスキル**: 異なる設計オプションを提示し、それぞれのメリット・デメリットを説明し、与えられた条件下でなぜ一方を選ぶのかを説明する能力

---

### ステップ4: ボトルネックとスケーリングの機会を特定する

**目的**: システムが様々な条件下で動作でき、さらなる成長を支える余地があるかを検討する

#### 検討すべき質問
- [ ] **単一障害点（SPOF）はあるか？** ある場合、堅牢性と可用性をどう改善できるか？
- [ ] **データのレプリケーションは必要か？** レプリケーションする場合、全バージョンの一貫性はどの程度重要か？
- [ ] **グローバルサービスか？** マルチ地域データセンターの展開でデータの局所性を改善する必要があるか？
- [ ] **エッジケースはあるか？** ピーク時の利用やホットユーザーなど、パフォーマンスを劣化させるパターンは？
- [ ] **ユーザー10倍増にどう対応するか？**

#### スケーリングに役立つ概念

| 概念                    | 説明                   |
| --------------------- | -------------------- |
| 水平シャーディング             | データを複数のサーバーに分散       |
| CDN（コンテンツデリバリーネットワーク） | 静的コンテンツをユーザーの近くから配信  |
| キャッシュ                 | 頻繁にアクセスされるデータを一時的に保存 |
| レートリミティング             | リクエスト数を制限してシステムを保護   |
| SQL / NoSQLデータベース     | 用途に応じたデータベースの選択      |

---

### ステップ5: レビューとまとめ

**目的**: 設計全体を振り返り、総括する

- [ ] 主要な設計判断とその根拠をまとめる
- [ ] 空間・時間計算量のトレードオフを説明する
- [ ] 設計がすべての要件を満たしているか確認する（面接全体を通して定期的に行うべき）
- [ ] さらなる改善の方向性を提示する

## 面接の時間配分（目安: 45分）

| フェーズ | 目安時間 |
|---------|---------|
| 自己紹介 | 5分 |
| ステップ1: 問題空間の定義 | 5〜10分 |
| ステップ2: ハイレベル設計 | 10分 |
| ステップ3: 深掘り | 10〜15分 |
| ステップ4: ボトルネック・スケーリング | 5〜10分 |
| ステップ5: まとめ + 質問 | 5分 |
